    Evaluating the Evaluation Board

    Now that we have a Discovery board application working, let's move to the Evaluation board.

    The STM3240G-EVA board is a complicated piece of gear, with Ethernet, RS232, CAN, and LCD.  Its file tree is STM32F2-F4_Demonstration_Builder_V1.0.1, which contains 370 .C files You will only need around 200 of those to compile the correct image for this board - I think (more on that below).  They cost around $350:

    http://www.digikey.com/product-search/en?x=0&y=0&lang=en&site=us&KeyWords=stm3240g-eval

    The board comes loaded with a touchscreen evaluation program that exercises all the peripherals (except maybe CAN, I have not spotted a screen for that).  If you need to refresh your board, binaries are distributed in the source release:

    ARM32\STM32F2-F4_Demonstration_Builder_V1.0.1\Project\Binary\STM324xG_EVAL_Demonstration_Builder_V1.0.1.hex

    I think there may be some minor behavior differences between the distributed binary and the one that comes with the board, but have not been able to develop any evidence.

    This article is being written about two months after the last one.  I spent about a week getting the build to run, then got occupied with a rush project.  Now that I am finally trying the old build on the EVAL hardware, I am finding that it does not work: shortly after reset, the MCU Register Window PC indicator gets stuck at address 0x08013bae, which is the Hard Fault Handler in the load map:


    So the next few articles will show how to overcome this kind of error.  The build will be explained when I'm sure it's right.

    Build Details: Source Changes

    In the ZIP for the IO_Toggle project, there are two code files outside my GCC-MAKE build folder.  They hold the minimum code changes I could get away with.  Those changes are conditioned by the GCC_MAKE define I added to the Makefile, so they should not keep you from using commercial tools or demos if you want to try them.

    stm32f4_discovery.h

    This file is in ARM32\STM32F4-Discovery_FW_V1.1.0\Utilities\STM32F4-Discovery if you are following my project structure.  I added #includes for all the header files in the Standard Peripheral Library.  That's a brute force approach that some will not appreciate, but I got tired of tracing individual compile errors and including individual headers.  The header files do not increase code size - you do that yourself using the THUMB_SOURCE list in the Makefile.  Having all those includes active just makes it easier to change that list.

    With this many header files in every build, there is a small risk of constant definitions overriding other constant definitions, but gcc will throw an error if a name is defined in more than one place.

    main.c

    This file includes a new, empty assert_param function and conditions in the existing assert_failed function.  Both are needed to complete the link  process, and are conditioned to the new GCC_MAKE definition in the Makefile.  There is a macro version of assert_param in stm32f4xx_conf.h, but it does not get expanded in library files that do not include that header.  The assert_param in my example clearly needs work, but it does allow the project to link.

    Build Details: Startup File and Loader Script

    When we built our first application in gcc, I showed the Makefile, but glossed over a couple of very important files.

    Loader Script File

    When gcc links object modules (.o or .obj files) for execution in an operating system, it leaves the addresses undefined.  Those addresses are allocated each time the executable is loaded by the operating system.  When gcc links embedded code, it must take the further step of setting up those addresses, i.e., loading.  Our Makefile uses the load script stm32f4xx_flash.ld to set up the addresses of the objects it links together.

    To me, this was the most risky choice in the whole build setup.  I normally rely on files like this to tell me how things are put together.  Coming in cold, and trying to set up my own builds, I knew that a mistake here would be both bad and easy to make.  Further, I had some initial trouble picking things for STM32F2 instead of STM32F4.  If you search the project structure I show here, all the other loader files are called "stm32_flash.ld".

    The loader file used here is from the RIDE7 template included in the DSP version of the Standard Peripheral Library, which we are not looking at yet.  I picked it mainly because it had my specific target chip in the name, so I thought it would be lower risk than the alternatives.

    Assembler Startup File

    When the processor resets, the first word of code it executes is not in main().  Like any other embedded system, it needs to set itself up, start its clocks and timers, check memory, and do other things to get ready to run main().  Our Makefile specifies that startup_stm32f4xx.s will be the startup code, and that file is included in the ZIP for that first project.

    Finding the right startup file was a bit of an adventure. 
    • It has to have the correct information for our chip
    • It has to have the right function call interface for our code structure, calling SystemInit() in system_stm32f4xx.c before main().
    • It has to be written in the correct language idiom for the gcc assembler.
    In earlier efforts, I tried a few startup files and settled on the one provided by ST for the RIDE7 toolchain, as you can see if you read the header comments in the zipped file.  Since the file is provided by ST, I don't think we are violating Raisonance's rights by using it here.  Other alternatives are provided in the CMSIS tree.  Look at this path if you are following my project structure:

    ARM32\STM32F4-Discovery_FW_V1.1.0\Libraries\CMSIS\ST\STM32F4xx\Source\Templates

    As always, make sure you get STM32F4xx and not F2xx or F1xx.  I also try to avoid "stm32" things with no other modifiers.

    Loading the Discovery Board with ST-Link Utility

    The last post showed how to connect the Discovery board to the PC and get the ST-Link driver going.  Now we will load the code we compiled in gcc. 

    1.  Start the ST-Link utility.  You should have gotten a new desktop icon when you installed that software.

    2.  Observe your Discovery board.  If the original software is still running on the STM32F4, the LEDs will be blinking.

    3.  Click Connect - the toolbar icon looks like an AC wall plug.  It will take about 3 seconds.  A progress bar pops up and goes away.  At the end, your display will look like this and your target processor (the STM32F4) will be halted (lights not blinking).


    4.  Look at some things on the Discovery board.  If your LEDs were blinking before, they should have halted.  A red-green debug activity light LD1 is changing color. 

    5.  On the ST-Link Utility, the chip information block at top right is filled in, and shows the Device ID 0x413.

    Important Note: In this tool, you have several places you can enter addresses and other values.  If you mean to enter hex, you have to provide the "0x" prefix.  This software really does what it's told.

    6.  Notice the Menu Bar.   The Target menu is your debug interface to the STM32F4 Target processor.  In that menu, you will load code and do a surprising number of other functions:


    7.  I've found the MCU Core option to be especially helpful.  That leads to a Registers dialog, where you can Halt, Step and Run the processor, and view registers.  I had fun for a minute making my LED blink pattern stop and start, and trying to Halt the processor when all the LEDs were on.


    8.  To load your IO_Toggle code, use menu File, Open File (Ctl O).  Find the IO_Toggle example we just compiled.  I think Hex or Bin formats will both work.


    9.  When you Open the file, you get a repeat of the Connect progress bar.  At this point, the file is loaded in your GUI but not into your board.

    10.  To load to the board, use menu Target, Program&Verify (Ctl P).


    11.  Click "Start" to start programming.  This should take 3-5 seconds.  It's not obvious, but the debugging services are doing a few things behind the scenes here.  Stub code is set up in the Target's RAM that moves bytes to its Flash, and then your program is set into Flash by the Target's processor core.  This is important to note because its not instant.  When you start using higher level tools, you may see a claim of success after a near-instant operation.  I have seen it even on expensive commercial toolchains.  No code download will be faster than the one you see here.

    12.  If you left the "Reset after programming" box clicked, your target will start blinking LEDs in a pattern after the code load is complete.

    If you want to make sure it's your code that's running, go to your main.c and change the blink pattern.

    Tip: if you press the black Reset button while debugging, you may need to Connect again.

      Connecting to the Discovery Board

      Now that we have some code to try, the next step is to connect to the Discovery board.  It does not come with a CD, so you have to know a couple of things to make this work.  The debug front end for the Discovery board is ST-Link firmware, which translates USB to JTAG for debugging the main processor.  On the PC side, ST-Link has a nice utility that lets you load code and inspect registers.  ST-Link is currently supported in its own page, not linked from the Discovery board page.  If that link goes stale, try googling "st-link site:st.com".

      Click the "Design support" tab and scroll down to software resources, and you find three files:
      • ST-Link Utility - PC side loader with register inspection.  It can also update the ST-Link firmware on the Discovery board.
      • ST-Link Firmware Upgrade -  I was confused by this at first, but it turns out this is the firmware for the USB-JTAG translation processor.
      • ST-Link USB Driver - this can cause problems, especially on 64 bit OSs.
      Before going any further, let me tell you what I did at first - I plugged in the Discovery board and let Windows Update do its thing.  This seems to work for some people and not for others.  In my case, running Win7x64, it failed pretty thoroughly, showing "Device driver software was not successfully installed":


      The answer was given in the ST forum post linked above:
      • Get the files above
      • Make sure the Discovery board is not plugged in
      • Install the PC and Driver software packages
      • Plug in the Discovery Board
      • If the ST-Link is not detected, open Device Manager and Scan for Changes
      • If it looks like the driver load is stalling, make Windows stop searching online for drivers
      • If it looks like the install worked, open the ST-Link Utility and Connect to the Discovery board (the toolbar icon looks like an AC wall plug).
      This just worked for me.  I remember having more trouble before.  I think I may have had the Discovery board plugged in while installing the software.

      Connection note: the Discovery board has two connectors, one at each end.  These exercises use the USB connection on the end with the ST-Link chip, which is the STM32F103 that runs the "firmware."

      The IO_Toggle Makefile

      This is the Makefile from the last article with comments.  Do not try to copy&paste this text - the comments will mess you up.  Get the code files instead.

      Relative path to the ARM32 top level development folder:
      BASE_DIR=../../../../..



      Relative path to the various library folders:
      CMSIS_DIR=$(BASE_DIR)/STM32F4-Discovery_FW_V1.1.0/Libraries/CMSIS
      LIB_DIR=$(BASE_DIR)/STM32F4-Discovery_FW_V1.1.0/Libraries/STM32F4xx_StdPeriph_Driver
      BOARD_DIR=$(BASE_DIR)/STM32F4-Discovery_FW_V1.1.0/Utilities/STM32F4-Discovery

      Point to the CodeSourcery toolchain:
      CC=arm-none-eabi-gcc
      OBJCOPY=arm-none-eabi-objcopy


      Point to the linker script:
      LDSCRIPT=stm32f4xx_flash.ld

      Point to emacs' etags, which is not in the path:
      ETAGS=/cygdrive/c/emacs/emacs-23.4/bin/etags.exe


      Set up the compile and link:
      LINKER_FLAGS=-mthumb -nostartfiles -Xlinker -oLibdemo.elf -Xlinker -M -Xlinker -Map=libdemo.map
      LIBS=

      DEBUG=-g
      OPTIM=-O3


      CFLAGS= $(DEBUG) \
              $(OPTIM) \
              -T$(LDSCRIPT) \

      Set up the include folders:
              -I . \
              -I .. \
              -I $(CMSIS_DIR)/ST/STM32F4xx/Include \
              -I $(CMSIS_DIR)/Include \
              -I $(LIB_DIR)/inc \
              -I $(BOARD_DIR) \


      Set up definitions that go in each compile as if it were a #define in the code:
              -D GCC_ARMCM4F \
              -D STM32F4 \
              -D THUMB \

      Create a special #define just for this compile:
              -D GCC_MAKE \


      Not sure we need this:
              -D USE_FULL_ASSERT \


      These are chip specific instructions that took me a long time to tweak out:
              -mcpu=cortex-m4 \
              -D PACK_STRUCT_END=__attribute\(\(packed\)\) \
              -D ALIGN_STRUCT_END=__attribute\(\(aligned\(4\)\)\) \
              -fomit-frame-pointer \
              -mthumb \
              -mfloat-abi=softfp \
              -fno-strict-aliasing \
              -fno-dwarf2-cfi-asm

       Sources.  Some of the library code could be trimmed.  Thumb is 16 bit microcode that sames ROM space:
      THUMB_SOURCE= \
              ../main.c \
              ../stm32f4xx_it.c \
              ../system_stm32f4xx.c \
              $(BOARD_DIR)/stm32f4_discovery.c \
              $(LIB_DIR)/src/misc.c \
              $(LIB_DIR)/src/stm32f4xx_dcmi.c \
              $(LIB_DIR)/src/stm32f4xx_dma.c \
              $(LIB_DIR)/src/stm32f4xx_exti.c \
              $(LIB_DIR)/src/stm32f4xx_flash.c \
              $(LIB_DIR)/src/stm32f4xx_fsmc.c \
              $(LIB_DIR)/src/stm32f4xx_gpio.c \
              $(LIB_DIR)/src/stm32f4xx_rcc.c \
              $(LIB_DIR)/src/stm32f4xx_rtc.c \
              $(LIB_DIR)/src/stm32f4xx_spi.c \
              $(LIB_DIR)/src/stm32f4xx_syscfg.c \
              $(LIB_DIR)/src/stm32f4xx_tim.c \
              $(LIB_DIR)/src/stm32f4xx_usart.c \
              $(LIB_DIR)/src/stm32f4xx_wwdg.c \

      ARM Source is normal 32 bit instructions:
      ARM_SOURCE=
      This is regular makefile stuff:
      THUMB_OBJS = $(THUMB_SOURCE:.c=.o)
      ARM_OBJS = $(ARM_SOURCE:.c=.o)


      all: Libdemo.bin

      Libdemo.bin : Libdemo.hex
          $(OBJCOPY) Libdemo.elf -O binary Libdemo.bin

      Libdemo.hex : Libdemo.elf
          $(OBJCOPY) Libdemo.elf -O ihex Libdemo.hex
       

      The Startup assembler file is very important.  You have to pick the right one out of all the examples.  This one works:
      Libdemo.elf : $(THUMB_OBJS) $(ARM_OBJS) startup_stm32f4xx.s Makefile
          $(CC) $(CFLAGS) $(ARM_OBJS) $(THUMB_OBJS) $(LIBS) startup_stm32f4xx.s $(LINKER_FLAGS)


      $(THUMB_OBJS) : %.o : %.c Makefile
          $(CC) -c $(CFLAGS) -mthumb $< -o $@

      $(ARM_OBJS) : %.o : %.c Makefile
          $(CC) -c $(CFLAGS) $< -o $@

      clean :
          rm $(THUMB_OBJS)
          touch Makefile
      Make tags for emacs:
      tags:
          rm -f TAGS
          echo $(THUMB_SOURCE) $(ARM_SOURCE) | xargs $(ETAGS) -a -o TAGS
          ls $(CMSIS_DIR)/ST/STM32F4xx/Include/*.h | xargs $(ETAGS) -a -o TAGS
          ls $(CMSIS_DIR)/Include/*.h | xargs $(ETAGS) -a -o TAGS
          ls $(LIB_DIR)/inc/*.h | xargs $(ETAGS) -a -o TAGS
          ls $(BOARD_DIR)/*.h | xargs $(ETAGS) -a -o TAGS

        Discovering the Discovery Board

        Looking at the code for the Eval board, it looks like a big job getting that first compile to run.  So let's take a step back and look at the "STM32F4 Discovery Board" (google that string with "site st.com" if the link goes stale).  This is a really amazing little board, available widely for around $15 (Digikey, Mouser, others).  Like the larger Eval board, it supports JTAG debugging over USB.  It has 4 LEDs, one user button, a MEMS accelerometer and a lot of breakout pins.

        It may even be possible to use the STM32F4 discovery board as a USB-JTAG interface for other boards.  I really hope to explore that further in the future.

        On the "Design support" tab of the ST page, scroll down to Firmware and find:

        http://www.st.com/internet/com/SOFTWARE_RESOURCES/SW_COMPONENT/FIRMWARE/stm32f4discovery_fw.zip

        That zip contains STM32F4-Discovery_FW_V1.1.0, which in turn contains STM32F4xx_StdPeriph_Driver V1.0.0 / 30-September-2011.  We'll use that library to start with, but will soon rebase to the newer version from the Eval release.


        The Project folder of this Release is different from the Eval board's.  Instead of one main application, there are a large number of individual peripheral demos.  We'll see later that there are issues with combining these demos - there is a programmable peripheral clock that is set differently in different examples.  But for now we can start with the simplest of the Peripheral Examples - IO_Toggle.

        Before moving on, a word about the project files:
        • main.c: contains main() of course, along with an assert handler that will become important later, and a busywait Delay function.
        • stm32f4xx_it.* - Interrupt handlers.  These files contain a surprising amount of application code in other samples, but in this one they mainly hold empty interrupt handlers.  Note that the default interrupt handlers are declared as "weak" names, so any instance of the same name in any link operation overrides the default handler.  The interesting names all end in "_Handler", so there is little chance of accidentally overriding an interrupt handler unless you happen to like calling things *_Handler().
        • system_stm32f4xx.c - chip setup routines called from assembler before main().
        • Compiler specific build folders - contain configuration files for the various toolchains.  In FreeRTOS, the compiler specific folders often contain startup code that conflicts with what's released by ST, but we don't have that problem at this level.
        This structure is consistent across most ST Releases.

        Inside an ST Software Release (1)

        In the last note, we found the demo firmware for the high end STM32F4 demo board, which has LCD, Ethernet, CAN, USB, LEDs, buttons and a memory expansion bus.  Below is my development folder, showing a Download directory and the unzipped copy of the main downloaded file.  The zip file name is not versioned, but the top level folder is.  We will be working with STM32F2-F4_Demonstration_Builder_V1.0.1, which contains:
        • Libraries - versioned drops of ARM CMSIS as well as ST supplied library code
        • Project - an unversioned drop of the application code.  I assume the version of the top level folder applies to the application code.
        • Utilities - versioned drops of the Eval Board Support Package, the Lightweight IP stack (lwip), a FAT library and other goodies.
        The Project code has a structure that you see a lot in ST releases:
        • Binary - so you can compare your compile with something they built
        • Config - C headers full of #defines that set up the various modules
        • Core and Modules - the application code in two layers
        • EWARM, MDK-ARM and TrueSTUDIO - build setups specific to certain compiler vendors, and only those vendors
        The compiler specific folders bother me, because if you don't have one of those compilers you have to improvise.  Further, I think we'll see as we proceed that different projects include support for different compilers.  If we can make it all work with gcc, we won't have to worry about it.

        Getting Started 2: Sample Code

        Like many programmers, I rarely start out with a blank editor.  I usually find working sample code, compile and try it, then start changing it.  By the time I'm done, I may have rewritten the whole thing.  But, especially with a new processor, I find it's best to learn from the experts.

        With STM32F4, I started out with google, searching things like "stm32f4 sample code."  There were plenty of hits, but many of the links were stale.  It looks like ST reorganized its support site soon after this chip was introduced. From reading forum posts and other blogs, it seemed that it was easy to find the right file if you knew what you were looking for, but pretty hard if you were trying to get started.  It also seems that people who buy commercial compilers have a leg up because ST code comes bundled with their compiler packages.  But I have seen cases where the commercial compiler had old code, and things would not work until you rebased to the latest ST release.  Same goes for FreeRTOS.

        In this article, I will try to show how to get and stay oriented on the ST support site, and find the best code for your work.

        ST Code Downloads
        Let's start by finding the code for the ~$300 Eval board, ST part number STM3240G.  As of this writing, the file you probably want is:

        http://www.st.com/internet/com/SOFTWARE_RESOURCES/SW_COMPONENT/FIRMWARE/stm32f2-f4_demobuild.zip

        But if ST moves it, that link will be stale - many links like the above are stale now from an ST website redesign this year.  To find the right thing in the future, google the keyword "STM3240G firmware," not "STM3240G sample code" or "STM3240G demo code."  Also, concentrate on the Eval board in your searches, not "STM32F4."  I went down some bad paths looking for "STM32F4 sample code," when "STM3240G firmware" would have taken me right to the money.  Who knew?

        For now, the best route is to google "STM3240G site:st.com", pick out the main product page, click the "Design Support" tab and go to the FIRMWARE heading at the bottom. That currently shows you two ZIP files.  The one I linked to is the Demonstration Builder Platform, which lets you duplicate the demo code shipped on the board, more or less.  The other just has an older version of the Binaries.

        In the next section, I will download both zips and examine what's inside.

        A Free ARM Cross Compiler for Windows

        ARM has been around so long that compilers for it have come and gone.  When you google, the top hits are often for old stuff.  The GNU-ARM project is inactive, with the last update in 2006.  The mainstream gcc code contains ARM support, and there are many tutorials for setting up your own gcc based ARM toolchain.  The Summon ARM toolchain is highly regarded, and this tutorial explains it well.

        But there are so many variants of ARM  and gcc that I was never sure I was working towards the right goal.  I felt like I was lost in a twisty maze of options on the order of arm-something-something-gcc.  Exploring any of the options would take nontrivial time, and I could not tell from the names which of the options were suitable.  With all respect to the summon-arm people, it was the last straw for me, because I could not imagine what "summon" might mean!

        In the end I went for the CodeSourcery toolchain, hosted by Mentor Graphics.  They have a commercial CodeBench, which I am sure works great.  But they also host a Lite edition, which means free command line gcc for ARM, precompiled.  It's recommended fairly widely in the community, and seems to do what you need done.  I hope in the future to loop back to old school gcc and learn how to set up my own toolchain.  But I am using CodeSourcery to get started.

        To find the toolchain, you can google CodeSourcery Download, and find the main CodeSourcery page.  Then "just" follow the links to the Lite edition - you will have to create an account to download code.  That link was here as of this writing, but if they move their pages around, you should be able to find it again using the path I described:

        http://www.mentor.com/embedded-software/sourcery-tools/sourcery-codebench/editions/lite-edition/

        Your choices under ARM are Symbian, Linux and EABI.  You want the EABI version.  This FAQ says why, and hints at what the different terms mean.  For the record, "An embedded-application binary interface (EABI) specifies standard conventions for file formats, data types, register usage, stack frame organization, and function parameter passing of an embedded software program".  The EABI we're using here is specified by the ARM company and is standard for all ARM compilers, but like everything else it has "flavors."

        After download,  I performed the Typical installation without incident, allowing the installer to set PATH for all users (on Win7x64).  Everything seemed ok at this point, though in my previous Linux efforts I ended up needing to add some files for standard C functions under FreeRTOS.

        After installation, you have access to the program arm-none-eabi-gcc.exe and associated utilities.  In that name, the "arm" and "gcc" should be pretty obvious, and "eabi" is explained above.  I'm still trying to figure out what the "none" means.  To check, open a command window and run "arm-none-eabi-gcc -v".

        First Surprise: Windows > Linux

        When I started working with the STM32F4, I assumed that I would be compiling with open source tools under Linux.  Almost all my previous ARM experience had been with embedded Linux platforms like Gumstix.  That kind of work is almost all Linux and gcc.  My main development system for several years was FC14 with XP running in a VM.  Since that time, my Windows work has expanded, so I now use a Win7x64 laptop every day.  So the first thing I did, before downloading any code, was to set up an FC17 VM. 

        For virtual machines, I favor VMWare Player.  The newer versions let you create and configure new VMs without Workstation.  After a few weeks of experimentation, the VM has grown to about 30 GB.

        It turns out that Windows would have been better.  ARM's demo and library software ships with programming variants for a few commercial toolchains, all based on Windows.  Plus, the free ST-Link utility is Windows based.  That program lets you do a surprising amount of work, including loading code, breaking execution and inspecting registers.  I actually went through a phase of compiling in Linux, copying to Windows and loading via ST-Link.  I think, going forward, I am going to try to do it all in Windows, with a little help from Cygwin.

        Getting Started

        In mid 2012, an old associate approached me with a project - a custom controller board based on the then-new STM32F4.  With CAN as well as UART, USB and Ethernet, it's a perfect chip for an industrial communications hub.  Its zero wait state execution from Flash and fast clock give it speed, and the DSP module enables analog and mixed mode applications.  My friend thought, and I agreed, that it was a perfect chip for his needs.

        The thing is, STM32F4 kicked my ass.  I thought I had pretty good ARM experience.  In 1998, I worked with an ARM core deeply embedded in an Ericsson PBX chip.  I've worked with embedded Linux on several ARM platforms including Gumstix (both Marvell and TI).  I figured, buy a development board, download some software, what can go wrong?  Earlier this year, my company did its first PIC32 project, and that's how it went.  I have done the same thing with Rabbit, Parallax Propeller, the SX family and others.  I knew that ARM processors were complicated, but was not prepared for the permutations of tool options, embedded modules, and library/demo code.

        Every time we hit a problem, I googled it.  I always found other people with the same problem, but rarely found a solution that worked for our group.  Our local ST support group was shocked to hear about our problems, but wasn't really able to help much either.  We started out with an open source toolchain, and at first thought that must have been the problem.  But then the client spent $10k on commercial tools, and the same problems kept happening.  I was starting to think something was seriously wrong, but this thread on LinkedIn showed me that ARM experts were taking the new chip in stride.

        As of this writing, the original project is in regroup mode, and I am trying to become an ARM expert.  Read along if you want.  No doubt some of this will seem childishly simple to some readers and inaccessible to others.  All I can say is that this blog will note the things that I find notable as I learn this chip.
